# Критичний аналіз та рекомендації щодо оцінки якості ArUco-маркерів

У цьому документі наведено критичний аналіз виконаних раніше завдань з оцінки якості бітових матриць ArUco-маркерів та надано детальні рекомендації щодо розробки нової програмної системи автоматичного визначення найкращих маркерів.

---

## 1. Критичний аналіз виконаних завдань

### 1.1 Аналіз властивостей маркерів та початкової евристики (`.tasks/marker_matrix_results.md`)
*   **Досягнення**: Було проведено важливий кореляційний аналіз (Пірсон та Спірмен) для виявлення геометрії маркерів, що впливає на розпізнавання (Detection Rate). 
  Змінна $r позначає коефіцієнт кореляції Пірсона (Pearson correlation coefficient), иа має значення межах від -1 (повна зворотна кореляція) до +1 (повна пряма кореляція), де 0 означає відсутність лінійного зв'язку.
  Це статистичний показник, який вимірює силу та напрямок лінійного зв'язку між двома змінними,
  а саме відображає зв'язок між певною геометричною властивістю бітової матриці маркера
  (наприклад, наявністю шахових візерунків 2x2) та успішністю його розпізнавання (Detection Rate).
  Виявлено сильний негативний вплив шахових візерунків 2x2 ($r = -0.668$), ізольованих клітинок ($r = -0.540$) та кількості переходів ($r = -0.513$), наприклад:
  - значення r = -0.668 вказує на сильний негативний зв'язок: зі збільшенням кількості шахових візерунків показник успішності розпізнавання маркера суттєво знижується;

*   **Недоліки**: 
    1.  **Лінійність евристики $Q$**: Початкова формула використовувала лінійну комбінацію ознак з фіксованими вагами ($w_1, \dots, w_5$). Це припускає повну незалежність ознак (що є хибним, оскільки кількість переходів $T$ безпосередньо корелює з наявністю ізольованих клітинок $I$).
    2.  **Відсутність жорстких меж**: Лінійна оцінка дозволяла маркеру з катастрофічним дефектом (наприклад, наявністю шахового візерунка 2x2) отримати прийнятний бал за рахунок ідеальних значень інших параметрів.
    3.  **Не враховано баланс кольорів**: Початкова модель не враховувала значущість балансу чорного та білого кольорів (Ones) та його вплив на розмиття чорної рамки маркера камери.

### 1.2 Оцінка програми (`utils/evaluate_markers.py`)
*   **Досягнення**: Скрипт успішно реалізував базове парсування маркерів, попіксельну сегментацію зображень маркерів для вилучення внутрішніх матриць та базові фільтри (premium, balanced, basic) для компенсації обмежень лінійної комбінації.
*   **Недоліки**: 
    1.  **Хардкодинг параметрів**: Ваги та порогові значення для фільтрації зашиті безпосередньо у коді або передаються через аргументи командного рядка CLI без будь-якої автоматичної калібровки.
    2.  **Обмеженість набору ознак**: Використовуються лише 6 первинних геометричних метрик. Більш складні просторові взаємодії та деталі контуру ігноруються.

### 1.3 Аналіз експериментів та ML-підходу (`.tasks/marker_matrix_results-2.md`)
*   **Досягнення**: Перехід до аналізу більшої вибірки маркерів (214 одиниць) та використання моделей машинного навчання (Random Forest, Logistic Regression) є принциповим кроком вперед. Було отримано реальні оцінки важливості ознак (Feature Importance): Chessboard (0.237), Ones (0.214), Transitions (0.186).
*   **Недоліки**:
    1.  **Відсутність інтегрованого пайплайну**: Незважаючи на теоретичну пропозицію розділити систему на етапи навчання (`train.py`) та оцінки (`rank_markers.py`), цей пайплайн не був повністю консолідований у готову до використання виробничу утиліту.
    2.  **Необхідність розширення простору ознак**: Було визначено потребу в додаванні таких метрик як Perimeter, EdgeWhiteCells, Components, але код для їх обчислення не був реалізований.

## 2. Рекомендації щодо створення нової програми визначення ID найкращих маркерів

Для створення надійної та науково обґрунтованої системи ранжування маркерів пропонується реалізувати нову програму зі структурованою модульною архітектурою.

### 2.1 Нова архітектура системи
Програма має бути розділена на такі модулі:
1.  **`features.py` (Модуль обчислення ознак)**:
    *   Реалізує вилучення матриці бітів з OpenCV ArUco генератора.
    *   Обчислює розширений набір ознак для 4x4 (або будь-якої іншої розмірності) бітової матриці:
        *   **Ones** (кількість білих клітинок).
        *   **OnesDev** (відхилення від ідеального балансу 50/50: $|Ones - \frac{N}{2}|$).
        *   **Transitions** (кількість переходів по горизонталі та вертикалі).
        *   **Chessboard** (кількість шахових підматриць 2x2).
        *   **Isolated** (кількість ізольованих 1x1 клітинок).
        *   **Cmax** (максимальний розмір зв'язної компоненти будь-якого кольору).
        *   **Homogeneous** (кількість повністю однорідних рядків або стовпців).
        *   *Нова ознака*: **Perimeter** (довжина межі між чорними та білими внутрішніми областями).
        *   *Нова ознака*: **EdgeWhiteCells** (кількість білих клітинок, що безпосередньо межують із зовнішньою чорною рамкою маркера — критично для розпізнавання контуру).
        *   *Нова ознака*: **ConnectedComponentsCount** (загальна кількість ізольованих зв'язних компонент).
2.  **`train.py` (Модуль навчання та калібрування моделей)**:
    *   Зчитує експериментальні дані (ідентифікатори маркерів та їх реальний Detection Rate).
    *   Формує цільову змінну $y = 1$, якщо $\text{Detection Rate} \ge 95\%$ (або інший цільовий поріг), та $0$ в іншому випадку.
    *   Навчає та оцінює за допомогою крос-валідації (5-fold CV) декілька моделей: **Logistic Regression** (для інтерпретації вагових коефіцієнтів) та **Random Forest / XGBoost** (для максимальної точності передбачення).
    *   Зберігає найкращу модель та скейлер ознак у файл серіалізації (наприклад, `best_marker_model.joblib`).
3.  **`evaluate.py` (Модуль ранжування маркерів)**:
    *   CLI-інтерфейс, який завантажує збережену ML-модель.
    *   Приймає на вхід діапазон або список ID маркерів, генерує їхні матриці бітів, обчислює ознаки та прогнозує ймовірність $Q = P(\text{Detection Rate} \ge 95\%)$.
    *   Сортує маркери за спаданням $Q$ та виводить детальний звіт у форматі CSV/Markdown.

### 2.2 Додатковий етап: Симуляційна валідація (Фізичне моделювання)
Для досягнення максимальної точності (золотий стандарт), рекомендується додати в нову утиліту режим **симуляційного тестування**:
*   Програмно рендерити маркер.
*   Застосовувати синтетичні фізичні спотворення: розмиття (Gaussian / Motion Blur), адитивний шум (Gaussian noise), дисторсію об'єктива, перспективні спотворення (нахили камери) та варіювання яскравості/контрасту.
*   Тестувати стійкість детектування через `cv2.aruco.detectMarkers`.
*   Підсумковий симуляційний Detection Rate використовувати як додаткову ознаку для навчання моделі або як фінальний верифікаційний показник.
