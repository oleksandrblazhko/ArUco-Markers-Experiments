# Persona 

Раніше ти створив програму - utils\calibrate_camera.py

Але експерименти показали проблему:
1) програма показує гарні результутати калібрування
2) програма-клієнт aruco_client використовує результати калібрування, але демонструє погані результати візуалізації

Треба покращити програму калібрування.

Пропонуються наступні рекомендації.

## Рекомендація 1. Контроль покриття поля зору
Для кожного знімка:
center_x = np.mean(corners[:,0,0])
center_y = np.mean(corners[:,0,1])
зберігати центри.
Після збору всіх кадрів:
xs = [c[0] for c in centers]
ys = [c[1] for c in centers]
обчислити:
coverage_x = (max(xs)-min(xs))/image_width
coverage_y = (max(ys)-min(ys))/image_height
і вивести:
Coverage X = 82%
Coverage Y = 76%
Критерії

Добре:
Coverage X > 70%
Coverage Y > 70%

Погано:
Coverage X < 50%
або
Coverage Y < 50%

## Рекомендація 2. Контроль нахилів дошки

Багато користувачів рухають дошку лише по площині.
Для оцінки дисторсії потрібні різні кути.
Для кожного кадру можна оцінити:
rvec
після попереднього solvePnP.

Або простіше:
взяти співвідношення сторін знайденого чотирикутника.

Виводити підказки:
Need more tilted views
або
Rotate board around X axis
Rotate board around Y axis
Рекомендація 3. Контроль розміру дошки

Дошка повинна бути:
дуже близько
середньо
далеко

Зараз це не перевіряється.

Для кожного кадру:
area = cv2.contourArea(board_polygon)
або
bounding_box_area

Зберігати:
board_area_ratio =
board_area / image_area

Добре:
мінімум 15%
максимум 70%

Якщо всі кадри мають
30%-35%
то калібрування буде гіршим.

## Рекомендація 4. Перевірка країв кадру

Для ArUco це особливо важливо.
Подивіться на ваші попередні експерименти.
Проблеми з маркерами виникали:
біля правого краю;
біля лівого краю;
у кутах.

Тому корисно перевіряти, чи були кути дошки в усіх областях кадру.

Розбийте кадр на сітку:
+---+---+---+
| 1 | 2 | 3 |
+---+---+---+
| 4 | 5 | 6 |
+---+---+---+
| 7 | 8 | 9 |
+---+---+---+

та підрахуйте, в яких зонах був центр дошки.

Після збору:
Visited cells: 8/9
або
Visited cells: 4/9

## Рекомендація 5. Найкорисніша перевірка для вашого проєкту

Після калібрування зробити незалежний тест.

Зберегти параметри.
Показати користувачу:
Validation phase
Попросити розмістити дошку у:
центрі;
лівому верхньому куті;
правому верхньому;
лівому нижньому;
правому нижньому.

Для кожного положення:
solvePnP(...)
projectPoints(...)

та виміряти похибку.

Тоді можна отримати:

Calibration RMS = 0.18 px

Validation:
Center      = 0.21 px
Top-left    = 0.35 px
Top-right   = 0.42 px
Bottom-left = 0.38 px
Bottom-right= 1.74 px

І відразу видно проблему з правим нижнім кутом.

Для задачі позиціонування ArUco це набагато корисніше, ніж середня похибка репроєкції.

Після завершення калібрування виводити приблизно такий звіт:

=== CALIBRATION QUALITY REPORT ===

Reprojection error: 0.21 px

Coverage:
  X coverage: 84%
  Y coverage: 78%

Board scale:
  Min size: 18%
  Max size: 72%

Orientation diversity:
  X tilt: OK
  Y tilt: LOW

Grid coverage:
  8 / 9 regions visited

Overall:
  GOOD

Recommendation:
  Capture more tilted views near the right edge

Такий звіт значно краще передбачає реальну якість калібрування для системи на основі ArUco, ніж один лише reprojection_error. Особливо для вашого випадку, де камера жорстко закріплена над шаховою дошкою і критично важлива точність по всьому полю.
